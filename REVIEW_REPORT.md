# Agent Harness - Báo cáo Review Code tự động

## Kết luận: **🟢 APPROVE**

### Tóm tắt:
Panel review found no critical/high blockers, but identified several medium-risk robustness and data-integrity gaps around pre-commit staged content handling, watcher filesystem races, and setup/profile partial failures. Recommended fixes are mostly bounded error handling, idempotency guards, and explicit compensation for multi-step global setup operations.

| install_hooks.py | N/A | MEDIUM | error_handling | Staging a binary or non-UTF-8 text file can make the pre-commit hook fail unexpectedly when building staged file context/redacted diff. Concrete scenario: staging a binary file or a UTF-16 `.ps1`/`.json` file. | auto_watch.py | N/A | MEDIUM | race_condition | Filesystem churn during `snapshot()` can crash or produce noisy failures if a file disappears or becomes inaccessible between directory enumeration and stat/read. Concrete scenario: running the global watcher while `npm install`, tests, or temp-file churn creates and deletes files rapidly. If `Path.stat()`, `os.scandir`, or path normalization exceptions are not caught per entry, the watch loop can terminate despite the race being harmless. | Wrap per-file stat/path operations in `try/except (FileNotFoundError, PermissionError, OSError)` and skip that entry; use `os.walk(..., onerror=...)` or equivalent. Add a test that simulates one file raising `FileNotFoundError` during snapshot and verifies `snapshot()` completes. | tester, integrity |
| harness-full-setup.bat | N/A | MEDIUM | partial_failure_gap | The setup flow performs several persistent global side effects in sequence, including writing the runtime profile, copying `llmwiki/`, installing the `AgentHarnessAutoWatch` scheduled task, and optionally starting watcher/dashboard processes. If a later step fails after an earlier persistent step succeeds, the system can be left in a partially configured state, for example with startup automation enabled but setup reported as failed or smoke checks skipped. | Make setup steps explicitly idempotent and record completed steps in a small setup state/manifest. On failure, either roll back newly created scheduled tasks/profile changes where safe, or print a deterministic recovery command showing exactly what was changed and how to undo it. Add tests or scripted dry-runs covering failure after profile write and after scheduled-task creation. | integrity |
| harness-toggle.bat | N/A | MEDIUM | non_idempotent_operation | `Start-Watch()` and `Start-Dashboard()` start background processes as side effects of toggle/setup flows. Prefer delegating process ownership checks to the daemon itself, and return success when the desired process is already running. Add retry/concurrent invocation tests for watch/dashboard start commands. | integrity |
